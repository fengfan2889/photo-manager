# 项目开发规则

## 文档维护规则

### 开发日志更新（每次开发必须）

每次开发会话结束时，必须更新 `doc/开发日志.md`：
1. 在文件末尾添加新的开发记录
2. 记录日期时间、任务目标、文件变更、当前状态
3. 如有阻塞问题，在备注中说明

### 设计文档更新（架构变更时）

当发生以下情况时更新 `doc/设计文档.md`：
- 新增/删除核心模块
- 技术栈变更
- 架构调整
- 关键决策变更

## 当前工作文档

### 文档位置
`doc/当前工作.md`

### 规则
1. **开始新工作时**：在此文档记录当前工作内容，包括交互流程
2. **用户确认完成后**：清空此文档，记录"已完成"标记
3. **新工作开始时**：将交互记录写入此文档，包括：
   - 任务目标
   - 交互流程（用户请求 → 分析 → 实现 → 验证）
   - 文件变更
   - 验证结果

### 更新开发计划

当任务完成或计划调整时更新 `doc/开发计划.md`：
- 标记已完成任务
- 调整优先级
- 添加新任务

### 更新开发日志

每次开发会话结束时，必须更新 `doc/开发日志.md`：
1. 在文件末尾添加新的开发记录
2. 记录日期时间、任务目标、文件变更、当前状态
3. 如有阻塞问题，在备注中说明

**在开始任何开发工作之前，必须先完成设计文档的编写和人工确认。**

1. 用户需求 → 设计文档初稿
2. 多轮迭代评审（可使用 Oracle/Metis/Momus）
3. 人工确认
4. 开始开发

## 强制规则

| 规则 | 说明 |
|------|------|
| 🚫 禁止跳过设计 | 未经设计文档确认，不得开始编码 |
| 🚫 禁止擅自决策 | 关键技术选型必须经过评审和确认 |
| ✅ 必须多轮迭代 | 设计文档可能需要多次修改才能完善 |
| ✅ 必须人工确认 | 最终设计必须由用户明确批准 |
| ✅ 变更需重新确认 | 开发中设计变更需重新评审和确认 |
| ✅ catch 必须日志 | 所有的 catch 块除非用户指出，都要打印错误日志 |
